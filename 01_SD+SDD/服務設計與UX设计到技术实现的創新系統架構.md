## 問題意識與背景

### 核心問題的浮現

在設計早餐店訂餐系統的過程中，我們逐漸發現了一個關鍵的知識斷層：

> **服務設計的產出（藍圖 + User Stories）與技術實現之間，存在一道 UX 設計的鴻溝。**

### 具體困境

```
服務設計階段：
├── CJM：發現用戶排隊點餐耗時 5-10 分鐘
├── 藍圖：設計「手機預訂 + 到店即取」新流程
└── Story：「身為顧客，我想要預訂餐點...」
    ↓
    【斷層】不知道該如何轉化為具體界面
    ↓
技術開發階段：
├── 後端：直接開始設計 API
├── 前端：憑感覺畫界面
└── 結果：技術可用，但 UX 糟糕
```

### 關鍵提問

1. **User Story 是否直接對應網頁頁面？**
   - 初步假設：一對一
   - 實際發現：多對多關係

2. **服務藍圖的接觸點是否就是網頁頁面？**
   - 初步假設：是
   - 實際發現：接觸點是抽象概念，頁面是具體實現

3. **UX 設計在整個流程中扮演什麼角色？**
   - 發現：這是被忽視的關鍵轉化層

## 傳統方法的痛點分析

### 痛點一：服務設計直接跳躍到技術實現

```
傳統流程：
CJM → 藍圖 → Story ───────────────→ 開發
                        ↑
                        └── UX？被忽略或憑感覺做
```

**問題表現**：
- 工程師直接根據 Story 開始寫代碼
- 前端界面憑個人經驗設計
- 結果：能用但不順暢，用戶體驗差

### 痛點二：Spec-Kit 的 UX 能力缺口

```
預期：服務設計 ──[Spec-Kit]──→ 技術規格
實際：服務設計 ──[Spec-Kit]──→ 技術可行但 UX 糟糕的規格
```

**發現**：Spec-Kit 的 `/specify` 具備優秀的技術規格生成能力，但缺乏專業 UX 設計能力。

### 痛點三：設計與開發的語言斷層

| 角色 | 語言 | 斷層 |
|------|------|------|
| 服務設計師 | 藍圖、旅程、觸點 | 開發聽不懂 |
| UX 設計師 | 線框圖、原型、交互 | 常被跳過 |
| 開發工程師 | API、Schema、組件 | 不理解設計意圖 |

## 理論框架建立

### 核心洞察：三層轉化模型

經過深入討論，我們確立了以下框架：

> **服務設計的產出必須經過「UX 設計層」的轉化，才能有效進入「技術實現層」。**

```
┌─────────────────────────────────────────────────────────────┐
│                    服務設計層（抽象）                         │
│  • CJM：人的洞察，發現痛點與機會                              │
│  • 藍圖：系統視角，描述如何運作                               │
│  • Story：用戶視角，描述想要什麼                              │
└─────────────────────────────────────────────────────────────┘
                            ↓ 轉化
┌─────────────────────────────────────────────────────────────┐
│                    UX 設計層（橋樑）                          │
│  • 信息架構：如何組織內容                                     │
│  • 頁面流：如何在頁面間移動                                   │
│  • 交互設計：如何操作界面                                     │
└─────────────────────────────────────────────────────────────┘
                            ↓ 轉化
┌─────────────────────────────────────────────────────────────┐
│                    技術實現層（具體）                         │
│  • 前端：頁面、組件、路由                                     │
│  • 後端：API、Schema、數據庫                                  │
│  • 部署：服務器、域名、監控                                   │
└─────────────────────────────────────────────────────────────┘
```

### 關鍵論證過程

#### 論證一：User Story 與頁面的關係

**初步假設**：一個 Story 對應一個頁面

**反駁證據**：
- US-002「建立訂單」需要多個頁面（購物車 → 結帳 → 付款）
- US-001「瀏覽菜單」和 US-005「搜尋品項」可能共用 /menu 頁面

**結論**：
> **User Story 是「功能需求」，頁面是「實現載體」，兩者是多對多關係。**

#### 論證二：服務藍圖與頁面的關係

**初步假設**：藍圖的接觸點就是頁面

**反駁證據**：
- 藍圖的「前台接觸點」是跨渠道的（Web、App、實體店）
- 網頁頁面只是其中一種實現方式

**結論**：
> **服務藍圖描述「在哪裡接觸用戶」，UX 設計決定「如何用具體頁面實現這些接觸點」。**

#### 論證三：UX 設計的必要性

**提問**：能否跳過 UX 設計，直接從 Story 生成代碼？

**實驗驗證**：
- 嘗試用 Spec-Kit 直接生成：結果界面混亂、流程不合理
- 補充 UX 設計後再生成：結果可用性大幅提升

**結論**：
> **UX 設計是「設計意圖」到「技術實現」的必要轉化層，不可跳過。**

## 三層轉化架構

### 架構總覽

```
┌─────────────────────────────────────────────────────────────────┐
│  LAYER 1: 服務設計層（Service Design Layer）                     │
│  ─────────────────────────────────────────────────────────────  │
│  輸入：用戶研究、市場分析、業務目標                               │
│  輸出：CJM、藍圖、User Stories                                    │
│  角色：服務設計師、產品經理                                       │
│  工具：Miro、Figma、Mermaid                                       │
└─────────────────────────────────────────────────────────────────┘
                              ↓
                    【轉化點：洞察轉為設計】
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  LAYER 2: UX 設計層（UX Design Layer）★ 關鍵創新層               │
│  ─────────────────────────────────────────────────────────────  │
│  輸入：藍圖、User Stories                                        │
│  輸出：信息架構、頁面流、線框圖、交互說明                         │
│  角色：UX 設計師、交互設計師                                      │
│  工具：Figma、Sketch、Whimsical、Mermaid                          │
└─────────────────────────────────────────────────────────────────┘
                              ↓
                    【轉化點：設計轉為規格】
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  LAYER 3: 技術實現層（Technical Implementation Layer）           │
│  ─────────────────────────────────────────────────────────────  │
│  輸入：UX 設計文檔 + User Stories（增強版）                       │
│  輸出：前端代碼、後端 API、數據庫 Schema、部署配置                │
│  角色：前端工程師、後端工程師、DevOps                             │
│  工具：Spec-Kit、Cursor、Claude、Elysia、React                    │
└─────────────────────────────────────────────────────────────────┘
```

### 各層的詳細職責

#### Layer 1: 服務設計層

**核心問題**：
- 用戶是誰？（Persona）
- 他們現在有什麼痛點？（CJM As-Is）
- 我們想要創造什麼樣的新體驗？（CJM To-Be）
- 系統該如何支撐這個體驗？（藍圖）
- 具體要實現什麼功能？（User Stories）

**輸出物規範**：
```markdown
## 04-user-stories.md 標準格式

### Story ID: US-XXX
**身為** [角色]  
**我想要** [功能]  
**如此** [價值]

**驗收標準**:
- [ ] 可測試的條件 1
- [ ] 可測試的條件 2

**對應藍圖接觸點**: [前台接觸點名稱]
**優先級**: P0/P1/P2
```

#### Layer 2: UX 設計層 ★

**這是本架構的創新核心**

**核心問題**：
- 哪些功能放在同一頁？（信息架構）
- 用戶如何在頁面間流動？（頁面流）
- 每個頁面該長什麼樣？（線框圖）
- 如何交互？（交互說明）

**輸出物規範**：
```markdown
## 06-ux-design.md 標準格式

### 1. 信息架構（IA）
```
- 首頁 (/)
  - 推薦區
  - 快捷入口
    - 瀏覽菜單 → /menu
    - 查看訂單 → /orders

- 菜單頁 (/menu)
  - 分類導航（頂部固定）
  - 篩選器（橫向滾動）
  - 品項網格（2 列）
    - 點擊 → 詳情彈窗

- 品項詳情（彈窗 /menu/:id）
  - 圖片區
  - 規格選擇
  - 加入購物車按鈕
```

### 2. 頁面流（Page Flow）
```mermaid
flowchart LR
    A[/menu] -->|點擊品項| B[彈窗詳情]
    B -->|加入購物車| C[/cart]
    C -->|結帳| D[/checkout]
    D -->|完成| E[/order/success]
```

### 3. 關鍵交互說明
| 場景 | 用戶操作 | 系統反饋 |
|------|---------|---------|
| 加入購物車 | 點擊按鈕 | 1. 按鈕動畫 2. 購物車圖標+1 3. 不跳轉頁面 |
| 錯誤提示 | 提交失敗 | Toast 提示，保留表單內容 |
| 載入等待 | 進入頁面 | 骨架屏，避免白屏 |

### 4. 響應式斷點
- 桌面：> 1024px，側邊欄佈局
- 平板：768-1024px，調整網格列數
- 手機：< 768px，單列，底部固定導航
```

#### Layer 3: 技術實現層

**增強輸入**：不再只有 User Stories，而是 **User Stories + UX 設計文檔**

**執行流程**：
```
輸入：
  ├── 04-user-stories.md（功能需求）
  ├── 05-architecture.md（技術約束）
  └── 06-ux-design.md（UX 設計）★ 新增

工具：
  └── Spec-Kit（增強版）
      ├── /specify（讀取 UX 設計生成規格）
      ├── /plan（生成技術方案）
      ├── /tasks（拆分任務）
      └── /implement（生成代碼）

輸出：
  ├── 前端頁面（符合 UX 設計的佈局和交互）
  ├── 後端 API（支撐頁面數據需求）
  └── 數據庫 Schema（存儲頁面展示數據）
```

## 各層詳細設計

### 層間轉化規則

#### 轉化 1：服務設計 → UX 設計

**輸入**：藍圖中的前台接觸點 + Story 的功能需求

**轉化邏輯**：
```
接觸點「瀏覽菜單」
  ├── 功能需求：查看品項列表
  ├── 功能需求：搜尋品項
  ├── 功能需求：篩選分類
  └── 功能需求：查看品項詳情

轉化為：
  頁面：/menu
    ├── 模塊：分類導航（滿足篩選）
    ├── 模塊：搜尋欄（滿足搜尋）
    ├── 模塊：品項網格（滿足查看列表）
    └── 交互：點擊 → 彈窗（滿足查看詳情）
```

**決策依據**：
- 相關性功能聚合（分類、搜尋、列表都是瀏覽行為）
- 減少頁面跳轉（用彈窗代替新頁面查看詳情）
- 用戶認知負荷（一次只做一件事）

#### 轉化 2：UX 設計 → 技術規格

**輸入**：頁面流 + 線框圖 + 交互說明

**轉化邏輯**：
```
頁面：/menu
  ├── 模塊：分類導航
  │   └── 數據：分類列表（name, icon）
  │   └── API：GET /api/categories
  │
  ├── 模塊：搜尋欄
  │   └── 狀態：輸入關鍵詞
  │   └── API：GET /api/menu?q={keyword}
  │
  ├── 模塊：品項網格
  │   └── 數據：品項列表（id, name, price, image）
  │   └── API：GET /api/menu?category={id}
  │
  └── 交互：點擊品項 → 彈窗
      └── API：GET /api/menu/:id（獲取詳情）
```

**決策依據**：
- 每個數據需求對應一個 API 端點
- 模塊邊界決定組件拆分
- 交互邏輯決定狀態管理方式

### UX 設計層的具體方法

#### 方法 1：信息架構設計（IA）

**卡片分類法**：
```
將所有 User Story 的功能點寫在卡片上：
- 瀏覽品項列表
- 搜尋品項
- 篩選分類
- 查看品項詳情
- 選擇規格
- 加入購物車
- 查看購物車
- 修改數量
- 填寫取餐時間
- 選擇付款方式
- 確認訂單
- 查看訂單狀態

分類聚合：
├── 瀏覽相關（/menu）
│   ├── 瀏覽品項列表
│   ├── 搜尋品項
│   ├── 篩選分類
│   └── 查看品項詳情
│
├── 購物車相關（/cart）
│   ├── 查看購物車
│   ├── 修改數量
│   └── 選擇規格
│
└── 訂單相關（/checkout, /orders）
    ├── 填寫取餐時間
    ├── 選擇付款方式
    ├── 確認訂單
    └── 查看訂單狀態
```

#### 方法 2：頁面流設計

**用戶任務流程法**：
```
主要任務：完成一次訂餐

步驟分析：
1. 發現需求（推播/打開 App）→ 首頁
2. 瀏覽選擇（看菜單）→ /menu
3. 決定品項（看詳情）→ 彈窗
4. 確認加入（購物車）→ 首頁/彈窗關閉
5. 準備結帳（查看購物車）→ /cart
6. 填寫資訊（結帳）→ /checkout
7. 完成付款（確認）→ /order/success
8. 追蹤狀態（查看）→ /orders/:id

優化決策：
- 步驟 3 用彈窗而非新頁面（減少跳轉）
- 步驟 4 後留在 /menu（可繼續選購）
- 步驟 6-7 可合併（如果資訊少）
```

#### 方法 3：交互設計

**費茨定律應用**：
```
高頻操作放在拇指熱區：
- 加入購物車：右下角懸浮按鈕
- 查看購物車：底部導航固定
- 返回頂部：右下角（長列表時顯示）

低頻操作可遠離：
- 設置：個人中心內
- 幫助：側邊欄或底部
```

**米勒定律應用**：
```
分類數量 7±2：
- 頂部導航最多 5-9 個分類
- 超過則用「更多」收起

列表項 7±2：
- 一屏顯示 5-9 個品項卡片
- 超出則滾動加載
```

## 常見問題與解答

### Q1: UX 設計要花多少時間？

**A**: 取決於專案複雜度
- 簡單專案（3-5 個頁面）：1-2 天
- 中等專案（10-15 個頁面）：3-5 天
- 複雜專案（20+ 頁面）：1-2 週

**投資回報**：良好的 UX 設計可減少 50% 以上的後續返工。

### Q2: 沒有專業 UX 設計師怎麼辦？

**A**: 使用 AI 輔助 + 設計模式庫
- 參考競品（Benchmarking）
- 使用 Ant Design / Material Design 等設計系統
- AI 生成初版，人工審查調整

### Q3: Spec-Kit 未來會內建 UX 能力嗎？

**A**: 預測會有，但目前需自行補充
- 可能的發展：/ux-design 指令
- 可能的發展：整合設計模式庫
- 當前方案：人工 UX + Spec-Kit 技術實現

### Q4: 如何驗證 UX 設計的好壞？

**A**: 多層次驗證
1. **專家評審**：UX 設計師審查
2. **認知走查**：模擬用戶操作流程
3. **原型測試**：簡單原型給真實用戶試用
4. **數據驗證**：上線後分析轉化率、停留時間

## 結論

### 核心創新

本文檔提出了**三層轉化架構**的創新方法論：

> **服務設計（為什麼）→ UX 設計（如何體驗）→ 技術實現（如何構建）**

解決了傳統方法中「設計意圖」到「技術實現」的斷層問題。

### 關鍵洞察

1. **UX 設計不是可選項，而是必要橋樑**
2. **User Story 與頁面是多對多關係，需要 UX 設計決策**
3. **服務藍圖的接觸點需要 UX 設計轉化為具體頁面**
4. **Spec-Kit 需要 UX 文檔增強才能生成可用性高的產品**

### 實踐價值

- **對設計師**：明確了 UX 在 AI-Native 開發中的定位
- **對工程師**：提供了從設計到代碼的清晰路徑
- **對產品**：確保了技術實現符合用戶體驗設計意圖
