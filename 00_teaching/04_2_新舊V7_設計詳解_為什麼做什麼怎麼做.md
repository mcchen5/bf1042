# 新舊 V7 設計詳解：為什麼、做什麼、怎麼做

> 目的：把「舊 V7（歷史版本：JsonFile + userId:number）」與「新 V7（目前實作：userId:string + Auth/Store 分離）」的差異講清楚，讓升級到 V8/V9 時不會迷路。

---

## 1. 問題背景（為什麼要重建新 V7）

舊 V7 可以運作，但在「要繼續升級到 V8/V9」時，會出現三個長期成本：

1. 身分責任與資料責任綁在同一層

- `Store` 同時做 `login/getUserById` 與訂單資料存取。
- 一旦登入方案更動，`Store` 也要跟著改。

2. `userId:number` 對接 Better Auth 成本高

- Better Auth 常見是字串主鍵（例如 UUID 或 CUID）。
- 若 V7/V8 還維持 number，V9 會發生跨層型別翻轉。

3. 路由層耦合過多細節

- route 同時承擔認證與資料層決策，難以替換底層實作。

一句話：舊 V7 能跑，但不是「可長期演進」的地基。

---

## 2. 架構對照（做了什麼）

### 舊 V7（可運作版）

1. 核心特徵

- 儲存：JsonFile
- 身分：由 Store 直接處理（`login/getUserById`）
- ID：`userId:number`

2. 優點

- 上手快、容易 demo。
- 不需要外部服務即可跑全流程。

3. 限制

- Auth 與業務資料耦合。
- 升級到 PostgreSQL / Better Auth 時改動面積大。

### 新 V7（重建地基版）

1. 核心特徵

- 儲存：仍可先用 JsonFile（降低遷移阻力）
- 身分：抽出 `Auth` 模組
- ID：從起點就固定 `userId:string`

2. 具體改動

- 合約層：`User.id`、`Order.userId` 改為 string。
- 介面層：`Store` 移除 `login/getUserById`。
- 模組層：新增 `auth/Auth.ts`、`auth/DemoAuth.ts`、`auth/index.ts`。
- route 層：登入找 `auth`，訂單找 `store`。
- 資料層：`data/store.json` 已完成字串化（`users[].id`、`orders[].userId` 皆為 string）。

3. 效果

- V7 -> V8：可只替換 store 實作（Json -> Postgres）。
- V8 -> V9：可只替換 auth 實作（DemoAuth -> Better Auth）。

---

## 3. 重建順序（怎麼做）

建議固定使用以下順序，避免半套狀態：

1. 先改 contract truth

- 先把 `userId` 定義為 string，讓上游事實一致。

2. 再改 Store 介面

- 移除 auth 責任，只保留 menu/orders。

3. 補 Auth 抽象與實作

- 建立 `Auth` 介面與 `DemoAuth`。

4. 調整 Store 實作

- Json store 專注業務資料。
- 載入歷史舊資料時可把 number userId normalize 成 string（相容性策略）；目前主資料檔已是 string。

5. 最後改 backend route

- route 只做協調，不混入底層細節。

---

## 4. 舊 V7 到新 V7 的對應思考

1. 舊 V7 不是錯，而是階段任務不同

- 它完成了「先跑起來」的教學目標。

2. 新 V7 是演進型版本

- 目標是「未來可替換、可擴充、可升級」。

3. 兩者應該並存於教材

- 舊 V7：示範最小可行與快速回饋。
- 新 V7：示範邊界治理與長期演進。

---

## 5. 教學口訣

1. 先跑起來：舊 V7。
2. 要走遠：新 V7。
3. 想升 V8/V9：先把 V7 的邊界做乾淨。

---

## 6. 課堂驗收點

1. 能說出舊 V7 與新 V7 的責任邊界差異。
2. 能解釋為何新 V7 要先把 `userId` 固定成 string。
3. 能畫出 route -> auth/store 的依賴方向。
4. 能描述 V7 -> V8 -> V9 各階段只替換哪一層。

---

## 7. 與目前倉庫狀態對齊（避免誤解）

1. 本倉庫目前新 V7 的主資料檔 `data/store.json`，`users[].id` 與 `orders[].userId` 已是 string。
2. 教材中提到的 `userId:number` 是「歷史舊 V7 脈絡」，用來說明為何需要這次重建。
3. 課堂示範若看到 number -> string 正規化邏輯，屬於相容舊資料的保險設計，不代表主資料仍是 number。
