# 版本閱讀指南

本講義的目的很單純：幫助你在閱讀這個專案的「版本快照檔」時，不被目前主線 import 鏈混淆。

## 1. 為什麼需要這份指南（Why）

這個專案不是每一版都放在獨立資料夾，而是保留版本快照檔，例如：

- `backend_01.ts`
- `backend_02.ts`
- `backend_03.ts`
- `backend_04.ts`
- `backend_05.ts`
- `backend_06.ts`
- `store/json/JsonFileStore_01.ts`
- `store/json/JsonFileStore_02.ts`

這種做法有教學優勢，但也有副作用：

- 你看的是舊版檔案
- 但 import 可能連到目前主線（未加後綴）檔案
- 讀到一半就不知道自己在看「哪一版的真相」

所以要先建立一個習慣：

先確認版本，再對照檔案鏈，不要直接沿著 IDE 跳轉一路追到最新。

## 2. 現在的教學路線是什麼（What）

目前教材採雙路線：

1. 歷史對照線：`舊 V7 -> 舊 V8`
2. 正式主線：`新 V7 -> 新 V8 -> 新 V9`

補充：舊 V9 規劃已捨棄，不再作為課程目標。

## 3. 你要怎麼讀才不會亂（How）

### 基本規則

1. 看某一版時，優先追該版對應的版本檔。
2. 未加後綴的 `backend.ts`、`shared/contracts.ts`、`store/Store.ts`、`store/index.ts`，只在讀「目前主線」時使用。
3. 若正在看 V1-V6，先不要把未加後綴檔案當成該版基準。

### 快速警訊

若看到下面情況，先停一下回來查表：

- 檔名是 `_02`，但 import 到 `shared/contracts.ts`
- 檔名是 `_03`，但 import 到 `store/Store.ts`
- 講義正在講 V3，程式卻出現 V5 後才有的型別欄位

這通常不是你理解錯，而是版本閱讀路徑跳錯。

## 6. 各版本閱讀重點（為什麼、做什麼、怎麼做）

### V1-V2

- 為什麼：先證明流程可行，再解決資料不持久。
- 做什麼：把資料從 backend 內存抽到 JSON store。
- 怎麼做：引入 `Store` 介面與 `JsonFileStore`。

### V3-V4

- 為什麼：前端展示需求與部署現實問題浮現。
- 做什麼：升級菜單模型、修正圖片與 SPA fallback。
- 怎麼做：補欄位、加 normalize、做連續修正版。

### V5-V6

- 為什麼：進入使用者與 API 協作階段。
- 做什麼：加入登入與訂單生命週期，並把契約文件化。
- 怎麼做：`backend_05.ts` 完成功能，`backend_06.ts` 導入 OpenAPI。

### V7（新主線起點）

- 為什麼：把快照成果收斂為可維護主入口。
- 做什麼：以 `backend.ts` 作為實際開發/建置/部署主線。
- 怎麼做：README、scripts、測試全部對齊 `backend.ts`。

## 7. 最後建議

如果你的閱讀目標是「學演進」，不要一開始就只讀 `backend.ts`。

建議流程：

1. 先確認講義正在講哪一版。
2. 回這份指南查該版對應檔案。
3. 沿該版追蹤鏈讀完，再切下一版。

這樣才能真正看懂每一版「為什麼改、改了什麼、如何落地」。
